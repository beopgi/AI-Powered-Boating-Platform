# AI 기반 보팅 라이프 플랫폼 시스템 아키텍처 설계서

## 1. 문서 목적
이 문서는 **AI 기반 보팅 라이프 플랫폼**의 전체 시스템 구조를 제품/개발/운영 관점에서 일관되게 설명하기 위한 기준 문서입니다. MVP 구현 범위와 확장 방향을 동시에 다루며, 서비스 설계·개발·테스트·운영 의사결정의 공통 참조점으로 사용합니다.

## 2. 프로젝트 분석 요약
현재 프로젝트 문서(README, PRD, 와이어프레임, 아키텍처 다이어그램)를 종합하면 플랫폼의 핵심 방향은 다음과 같습니다.

- **모바일 퍼스트 올인원 경험**: 탐색(지도) → 추천 → 항해 기록 → 예약/결제 → 알림까지 단일 앱 흐름 제공
- **데이터 기반 개인화**: 사용자 행동, 위치, 항해 로그, 해양 공공데이터를 결합한 추천 고도화
- **운영 확장성**: 이벤트/예약 중심 MVP에서 커뮤니티·커머스·사업자 콘솔로 확장 가능한 모듈형 구조
- **신뢰성과 보안**: 인증/권한, 결제, 위치정보, 감사로그를 서비스 레이어에서 명확히 분리

## 3. 설계 목표
- **사용자 경험 일관성**: 앱 내 주요 여정(추천, 지도, 예약, 기록)을 빠르고 단순하게 연결
- **고가용성 API 운영**: 트래픽 급증 시에도 핵심 도메인 서비스의 안정성 유지
- **데이터 자산화**: 온라인 트랜잭션 데이터와 분석 데이터를 분리해 성능/확장성 확보
- **지속적 학습 구조**: 추천 모델의 피드백 루프를 배치·스트리밍 방식으로 운영

## 4. 시스템 컨텍스트
플랫폼은 Flutter 기반 모바일 앱(iOS/Android)을 클라이언트 채널로 사용합니다. 모든 요청은 API Gateway/BFF를 통해 유입되며, 인증을 통과한 요청만 도메인 서비스(사용자, 추천, 지도/POI, 항해 로그, 예약, 결제, 알림)로 분기됩니다.

데이터 계층은 목적별로 분리합니다.
- **OLTP DB**: 사용자, 예약, 결제, POI 메타데이터
- **Geo/시계열 저장소**: 항해 경로 및 위치 샘플
- **캐시(Redis)**: 빈번 조회 데이터와 추천 결과
- **Data Lake/Warehouse**: 행동 로그, 집계, 모델 학습 데이터

외부 연동은 지도 API, 해양 공공데이터, 결제 게이트웨이, 푸시 인프라(FCM/APNs)로 구성됩니다.

## 5. 논리 아키텍처 레이어

### 5.1 클라이언트 레이어
- Flutter 앱: 홈, 지도, 예약, 항해 기록, 마이페이지
- 오프라인 대응: 최근 POI/지도 타일/임시 로그 로컬 저장 후 재동기화
- 디바이스 센서 수집: GPS, 시간, 이동 속도

### 5.2 엣지 & 접근 레이어
- **API Gateway/BFF**: 라우팅, 버전 관리, 레이트 리미트, 요청 표준화
- **Auth Service**: OAuth/소셜 로그인, JWT 발급·재발급, RBAC 스코프 검증

### 5.3 도메인 서비스 레이어
- **User/Profile Service**: 계정, 선호도, 언어, 알림 설정
- **Recommendation Service**: 규칙 기반 + 행동 기반 하이브리드 추천
- **Map & POI Service**: 지도 탐색, 필터링, POI 상세 조회
- **Sailing Log Service**: 항해 세션, 경로 저장, 요약 통계
- **Events/Reservation Service**: 상품/좌석 재고, 예약 상태 전이
- **Payment Service**: 결제 승인/취소, 정산 이벤트 발행
- **Notification Service**: 푸시/인앱 알림, 리마인더, 장애 알림

### 5.4 데이터 & 인텔리전스 레이어
- 트랜잭션 저장소와 분석 저장소를 분리하여 성능 병목 최소화
- 이벤트 기반 수집 파이프라인으로 추천 모델 학습 데이터 축적
- ML 파이프라인(특징량 생성 → 학습 → 배포 → 온라인 추론) 운영

### 5.5 외부 연동 레이어
- Maps API(Google Maps/Mapbox)
- Marine Public Data(기상/해양 환경/안전)
- PG(Payment Gateway)
- 푸시 전송 인프라(FCM/APNs)

## 6. 핵심 데이터 흐름
1. **추천 흐름**: 홈 진입 → Gateway → Recommendation Service → 캐시/실시간 피처 결합 → 추천 응답
2. **지도 탐색 흐름**: 지도 이동/필터 적용 → Map & POI Service → 가용성 데이터 결합 → 결과 반환
3. **항해 기록 흐름**: 트래킹 시작 → 위치 샘플 적재 → 세션 종료 시 통계 생성 및 저장
4. **예약/결제 흐름**: 예약 시도 → 좌석 홀드 → 결제 승인 → 예약 확정 → 알림 발송
5. **피드백 루프**: 클릭/예약/이탈 이벤트 적재 → 학습 파이프라인 반영 → 모델 재배포

## 7. 비기능 요구사항(NFR)
- **성능**: 홈 추천 API P95 400ms 이하, 지도 검색 API P95 500ms 이하
- **가용성**: 핵심 API 99.9% 목표, 장애 시 읽기 전용/캐시 폴백
- **보안**: 전송/저장 암호화, 토큰 만료 정책, 감사로그 및 최소권한 원칙
- **확장성**: 서비스 단위 오토스케일, 비동기 이벤트 기반 결합
- **관측성**: 중앙 로그, 메트릭, 분산 트레이싱, SLO 기반 알람

## 8. MVP 및 확장 로드맵
- **MVP**: 인증, 추천(룰+기초 모델), 지도/POI, 항해 로그, 예약, 결제, 알림
- **P1**: 커뮤니티, 장비 예약, 커머스, 컨텍스트 기반 추천 고도화
- **P2**: 안전 가이드 AI 코파일럿, 사업자 운영 콘솔, 수요 예측/동적 가격
